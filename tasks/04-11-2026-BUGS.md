## P0 — HF hub cache ignored; STT model hits network on every startup ✅ FIXED

**Severity:** High (correctness + startup latency + breaks offline use).

**Symptom:** App appeared to "keep loading models from HuggingFace even though
they already exist locally." STT model (`whisper-large-v3-turbo-asr-fp16`)
made a network metadata request on every launch.

**Root cause:** `_resolve_model_path` in `src/models.py` only recognized the
LM Studio layout (`models_dir/org/name`). The user's STT model lived in the
HuggingFace hub cache (`~/.cache/huggingface/hub/models--org--name/snapshots/<rev>/`),
which does not match that pattern. The resolver fell through and returned
the HF repo ID. `mlx-audio`'s `get_model_path`
(`mlx-audio/mlx_audio/utils.py:140-147`) then called
`snapshot_download(repo_id, ...)` **without** `local_files_only=True` —
triggering a network metadata check on every startup to validate the
cached snapshot against the remote. Offline, this would fail entirely.

The TTS model (Kokoro) was unaffected because it *was* in LM Studio layout
and resolved to an absolute path on step 2.

**Fix:** Added a third resolution step between the LM Studio check and the
fall-through. `_resolve_from_hf_cache()` calls
`snapshot_download(model_name, local_files_only=True)`, which returns the
cached snapshot path with zero network traffic, or raises
`LocalEntryNotFoundError` to let us fall through to a real download. When
it succeeds, we hand `mlx-audio` an absolute path, short-circuiting its
downloader at the `model_path.exists()` check (`mlx-audio/mlx_audio/utils.py:129`).

**Resolution order is now:**

1. Absolute path → returned unchanged.
2. `models_dir / org / name` (LM Studio / custom layout) → absolute path.
3. HF hub cache via `snapshot_download(local_files_only=True)` → absolute snapshot path.
4. Not found → return the HF repo ID so `mlx-audio` will download it.

**Verified against the user's actual caches:**

```
TTS  → /Users/john/.lmstudio/models/mlx-community/Kokoro-82M-bf16
STT  → /Users/john/.cache/huggingface/hub/models--mlx-community--whisper-large-v3-turbo-asr-fp16/snapshots/624c19c9...
miss → mlx-community/this-model-does-not-exist-xyz (passthrough)
```

**Tests:** `tests/test_models.py` — 10 new unit tests covering all four
resolution branches, precedence ordering, directory-vs-file discrimination,
and the `local_files_only=True` flag contract on the HF cache helper.

**Files changed:** `src/models.py`, `tests/test_models.py` (new).

## P1 — Broken no-op field validators in `src/config.py` ✅ FIXED

**Severity:** High (correctness + dead code propagating).

All six `@field_validator(..., mode="before")` methods on `Settings` have
identical broken bodies:

```python
@field_validator("server", mode="before")
@classmethod
def coerce_server(cls, v: Any) -> Any:
    return v if isinstance(v, dict) else v
```

Both branches of the ternary return `v` — these validators do nothing. Four
were pre-existing (`server`, `tts`, `stt`, `limits`, `log`); this cycle
copy-pasted the same broken pattern for the two new sections, adding
`coerce_models` (`src/config.py:106-109`) and `coerce_dialogue`
(`src/config.py:111-114`).

**Why it matters:**
- Either they were intended to do real coercion (e.g. `else dict(v)` when a
  nested Pydantic model is passed in) and the logic is incomplete, or they
  should never have been written at all.
- Propagating a broken pattern without questioning it is the kind of thing a
  staff-engineer review would catch immediately. It signals the cycle
  prioritized "add lines that look like the neighbors" over understanding.

**Fix:**
1. Decide the intent. Given Pydantic v2 already coerces nested dicts into
   sub-models automatically, these validators are almost certainly unnecessary.
2. Delete all six validators (`coerce_server`, `coerce_tts`, `coerce_sst`,
   `coerce_models`, `coerce_dialogue`, `coerce_limits`, `coerce_log`).
3. Run `uv run pytest tests/test_config.py` to confirm TOML loading and CLI
   overrides still work end-to-end.
4. If any test fails, the validators were masking a real coercion need —
   implement it properly instead of restoring the no-op.

**Fix applied:** Deleted all seven `coerce_*` validators (`server`, `tts`,
`stt`, `models`, `dialogue`, `limits`, `log`) and removed the now-unused
`field_validator` import. Pydantic v2 already coerces nested dicts into
sub-models automatically, so the validators were pure dead code.

**Verified:** `uv run pytest` — 126 passed (11 in `tests/test_config.py`
confirming TOML loading, CLI overrides, and nested merging still work).

**Files:** `src/config.py`

## P3 — Eager model loading wastes startup time and memory ✅ FIXED

**Severity:** Medium (UX + resource waste on every invocation).

`src/cli/__init__.py:79-86` loads **both** TTS and STT for every subcommand
except `serve`:

```python
if ctx.invoked_subcommand != "serve":
    if not no_tts:
        mm.load_tts(settings.tts.model, models_dir=resolved_models_dir)
    if not no_stt:
        mm.load_stt(settings.stt.model, models_dir=resolved_models_dir)
```

Actual per-subcommand model needs:

| Subcommand  | Needs TTS | Needs STT |
|-------------|-----------|-----------|
| `speak`     | yes       | no        |
| `watch`     | yes       | no        |
| `transcribe`| no        | yes       |
| `listen`    | no        | yes       |
| `dialogue`  | yes       | yes       |
| `serve`     | (lifespan)| (lifespan)|

Only `dialogue` genuinely needs both. Every other subcommand loads a model
it will never use, costing multiple seconds of startup and gigabytes of
memory on each invocation. The `--no-tts`/`--no-stt` flags exist as a
workaround but require users to remember which command skips which model.

**Why it matters:**
- `cai speak "hi"` takes many seconds longer than necessary because Whisper
  is being loaded for nothing.
- The Unix-philosophy note in `CLAUDE.md` explicitly values "economy and
  elegance of design due to size constraints" — eager loading contradicts
  that.

**Fix (elegant approach):**
1. Declare each subcommand's model requirements at its module level, e.g.
   a `NEEDS_TTS`/`NEEDS_STT` attribute or a shared dataclass registry.
2. In the Click group, look up the invoked subcommand's requirements and
   load only what's needed, still honoring `--no-tts`/`--no-stt` as an
   explicit override.
3. Alternative (simpler): hardcode a small dict mapping subcommand name →
   (needs_tts, needs_stt) in `src/cli/__init__.py`.
4. Add a test that invokes each subcommand through the group and asserts
   only the required models were loaded.

**Files:** `src/cli/__init__.py`, each `src/cli/<cmd>.py`

**Fix applied:** Added a `MODEL_REQUIREMENTS` registry in `src/cli/__init__.py`
mapping each subcommand name to a `(needs_tts, needs_stt)` tuple:

```
speak      → (tts)
watch      → (tts)
transcribe → (stt)
listen     → (stt)
dialogue   → (tts + stt)
serve      → (neither — lifespan handles it)
```

The group callback now looks up the invoked subcommand and loads only what
that subcommand declares it needs, still honoring `--no-tts`/`--no-stt` as
explicit overrides. The special-case `if ctx.invoked_subcommand != "serve"`
branch is gone — `serve` is just another entry in the table.

**Tests:** `tests/test_cli_subcommands.py::TestLazyModelLoading` — 8 new
tests: a parametrized case for each of the 6 subcommands asserting only
the required models were loaded, plus two override tests confirming
`--no-tts` and `--no-stt` still suppress loading on `dialogue` (the one
command that needs both).

**Verified:** `uv run pytest` — **148 passed** (140 → 148).

## P6 — Debounce timer churn under rapid rewrites ✅ FIXED

**Fix:** Replaced the per-event `threading.Timer` with a single long-lived
worker thread driven by a `threading.Condition`. `on_modified` now just pushes
a deadline forward (`_pending_deadline = monotonic() + _DEBOUNCE_SECONDS`) and
notifies the condition — no Timer creation, no cancel-and-recreate. The worker
sleeps on `cond.wait(timeout=remaining)` until the deadline settles, then
issues one read. New events arriving mid-read queue up on the next pass.

Because `TextFileHandler` is now used by both `watch` and `dialogue` (post-P8),
the fix lands in both code paths for free. Added `handler.stop()` method and
call it from both commands' `finally` blocks so the worker exits cleanly
instead of leaking until process teardown.

**Tests:** `tests/test_cli_audio_io.py`
- Renamed `test_debounce_timer_coalesces_rapid_writes` →
  `test_debounce_coalesces_rapid_writes`; now polls for drain instead of
  joining a Timer.
- **New**: `test_single_worker_thread_across_many_events` — fires 500 events
  and asserts `threading.active_count()` grows by at most 1. Directly pins
  the "no thread-per-event churn" guarantee.

Full suite: 151 passing.

---

### Original report

**Severity:** Low (works correctly, but wasteful under pathological load).

`_SpeakFileHandler.on_modified` (`src/cli/dialogue.py:44-52`) and the same
pattern in `_TextFileHandler` (`src/cli/watch.py`) cancel and recreate a
`threading.Timer` on every filesystem event. Under a writer that modifies
the file hundreds of times per second we churn short-lived Timer threads
for no benefit — only the final one ever fires.

**Why it matters:**
- Cheap in the common case (one editor save = a handful of events), but
  not free. A pathological log-tailer or streaming writer would spin up
  thousands of Timer objects per second.
- The pattern is duplicated in two files (see P8), so any improvement
  should land in both.

**Fix:**
1. Replace the per-event Timer with a single long-lived worker thread fed
   by a `queue.Queue`. The watchdog handler drops a sentinel on the queue;
   the worker coalesces bursts with a short wait then calls `_read_and_speak`.
2. Or: keep a single Timer and only call `.cancel() + restart` when the
   event arrives *while the timer is pending* — don't recreate the Timer
   object, reuse a condition variable.
3. Option 1 is cleaner and composes better with shutdown (`shutdown.set()`
   wakes the queue worker immediately).

**Files:** `src/cli/dialogue.py`, `src/cli/watch.py`

## P7 — No barge-in: listener blocks on TTS mid-sentence ✅ FIXED

**Fix:** Wired a shared `barge_event: threading.Event` through both halves
of dialogue mode. `MicRecorder.record(on_speech_start=…)` now sets the event
on the VAD **rising edge** (the first chunk whose RMS crosses the threshold)
and clears it before returning, giving a live "user is currently speaking"
signal. `play_tts_streaming(..., cancel=…)` watches the event between
chunks, breaks the generator loop on set, and — critically — calls
`AudioPlayer.flush()` instead of `stop()` on the cancelled path.

The distinction matters: `AudioPlayer.stop()` waits for drain (all queued
samples play out), which would defeat the whole point of barge-in.
`AudioPlayer.flush()` already exists in mlx-audio and does exactly what
we need: clears the buffer and hard-stops the stream.

Dialogue plumbs a single `barge_event` into both `_make_speak_callback`
and `_listener_loop`. Keyword-only `cancel=` on `play_tts_streaming` and
default-None on the new params mean `speak`, `watch`, `listen`, and
`transcribe` are untouched.

**Tests:** `tests/test_cli_audio_io.py`
- `TestPlayTtsStreamingBargeIn` (3 tests): cancel set mid-stream queues
  exactly one chunk then `flush()` (not `stop()`); no cancel drains via
  `stop()`; `cancel=None` default behaves like the pre-P7 API.
- `TestMicRecorderBargeSignal::test_event_fires_on_rising_edge_and_clears_on_return` —
  a real `record()` invocation behind a fake `sounddevice.InputStream` that
  pumps silence → speech → trailing silence. Asserts the barge event was
  observed set **mid-pump** (not at end) and cleared before return.

`tests/test_cli_subcommands.py`
- `TestSpeakCallback::test_callback_forwards_barge_event_as_cancel` —
  the factory wiring reaches `play_tts_streaming(cancel=…)`.

**Not addressed here (follow-ups):**
- Echo/feedback loop (TTS output picked up by mic) — P12's concern.
  Current design will barge-in on TTS audio bleeding through the mic,
  which is *usable* on headphones and *chaotic* on open speakers. Needs
  either headphones or a "mute mic while TTS active + short recovery
  window" gate. Not in P7 scope.

Full suite: 156 passing (151 → 156).

---

### Original report

**Severity:** Medium (UX — dialogue mode feels unnatural).

In `cai dialogue`, if the user starts talking while TTS is mid-sentence,
`_listener_loop` calls `recorder.record()` fine, but then blocks on
`inference_lock` until the current TTS call finishes. By the time STT
runs, the user's reply may already be seconds stale, and TTS will have
talked over them.

**Why it matters:**
- True conversational feel requires either (a) interrupting TTS playback
  when VAD detects speech, or (b) ducking TTS volume while recording.
- The current behavior is serviceable for "dictate, then listen" workflows
  but fails the "interactive dialogue" promise of the subcommand name.

**Fix:**
1. Expose an `AudioPlayer.stop()` / `.cancel()` on the streaming TTS path
   (`src/cli/audio_io.py::play_tts_streaming`) so the listener can abort
   playback mid-stream.
2. Have `_listener_loop` signal a "user is talking" event as soon as the
   recorder crosses the VAD start threshold — not when recording completes.
3. In `_SpeakFileHandler._read_and_speak`, check that event between
   chunks and bail out cleanly if set.
4. Alternative: implement volume ducking via sounddevice gain control
   instead of hard stop. Less jarring but requires mixer plumbing.

**Files:** `src/cli/dialogue.py`, `src/cli/audio_io.py`

## P9 — Listener tight-loops on MicRecorder.record() failure ✅ FIXED

**Fix:** `_listener_loop` now tracks `consecutive_failures` and backs off
via `shutdown.wait(timeout=backoff)` (interruptible by Ctrl+C), doubling
from `_RECORD_BACKOFF_START` (0.5s) to `_RECORD_BACKOFF_MAX` (8.0s).
After `_RECORD_MAX_CONSECUTIVE_FAILURES` (10) it calls `shutdown.set()`
and exits with a user-visible "giving up" message — so a missing audio
device no longer melts a core, and a truly dead mic fails fast after
~30s instead of burning until Ctrl+C. The counter resets to 0 on any
successful `record()`, so transient glitches don't count against the cap.

`src/cli/listen.py` was checked and does **not** have the same pattern —
it doesn't catch the exception at all, so a record failure crashes the
command with a traceback. That's ugly but not a tight loop. Logged as a
follow-up (see note below); out of scope for P9.

New tests: `tests/test_cli_subcommands.py::TestListenerLoop`
- `test_record_failures_trigger_backoff_and_give_up` — 10 failures →
  shutdown set, wait sequence `[0.5, 1.0, 2.0, 4.0, 8.0, 8.0, ...]`.
- `test_record_failure_resets_backoff_after_success` — 3 fails, 1
  success, 1 fail → loop exits cleanly, counter reset verified.

Full suite: 150 passing.

---

### Original report

**Severity:** Medium (CPU burn + log spam on missing audio device).

`_listener_loop` (`src/cli/dialogue.py:97-102`) catches exceptions from
`recorder.record()` and `continue`s immediately:

```python
while not shutdown.is_set():
    try:
        audio_path = recorder.record()
    except Exception:
        logger.exception("MicRecorder.record() failed")
        continue
```

If the recorder raises *synchronously and immediately* (e.g. no audio
input device available, sounddevice PortAudio error), this becomes a
tight loop that spins at full CPU logging a traceback per iteration
until Ctrl+C. The same pattern on the STT generation catch (line 113-115)
is safer because the record call before it provides natural throttling.

**Why it matters:**
- Running `cai dialogue` on a headless box or with the mic unplugged will
  melt a CPU core and fill the log in seconds.
- The `listen` subcommand likely has the same pattern — check
  `src/cli/listen.py`.

**Fix:**
1. Add exponential backoff on repeated record failures: start at 0.5s,
   double up to a cap of ~5s, reset on success.
2. Or: fail fast — after N consecutive failures, `shutdown.set()` and
   exit the loop with a clear error message. Dialogue mode without a mic
   is useless anyway.
3. Option 2 is more honest; option 1 is more forgiving to transient
   device glitches.

## P10 — Observer watches entire parent directory ✅ FIXED

**Fix:** Ripped `watchdog` out of the runtime path entirely. `TextFileHandler`
is now a pure mtime-polling worker — its long-lived thread (from P6) `stat()`s
only the target file every `_POLL_INTERVAL` (0.3s) and calls `_read_new` when
size or mtime changes. No parent-directory observation, no `FileSystemEventHandler`,
no `Observer`, no cross-platform FSEvents/inotify/fsevents plumbing.

This is a much bigger simplification than the bug report suggested. The poll
interval *is* the debounce — rapid writes within one tick coalesce naturally
— so the P6 Condition/deadline machinery collapsed into a single
`Event.wait(timeout=_POLL_INTERVAL)` loop.

**Latency tradeoff:** ~150ms average (half a tick), ~300ms worst case, from
write to TTS. The pre-P10 watchdog path had ~0–50ms detect latency + 300ms
debounce, so end-to-end is actually comparable-to-slightly-better in the
common case. Acceptable for a human-speed TTS workflow; if someone needs
sub-100ms, tune `_POLL_INTERVAL` down.

**watch.py and dialogue.py commands** were both simplified:
- `watch.py`: no more `Observer()/schedule/start/stop/join` — just construct
  the handler and idle-wait until Ctrl+C.
- `dialogue.py`: main loop now joins on `listener_thread` (the only remaining
  non-main thread) instead of the observer. Shutdown sequence shortened to
  `shutdown.set() → handler.stop() → listener_thread.join(timeout=5)`.
- Dropped the `from watchdog.observers import Observer` import from both.

**Privacy + syscall win:** a `--speak-file /tmp/speak.txt` no longer causes
the process to wake on every file event in `/tmp`. We stat exactly one path,
exactly as often as we need to.

## P13 — Dialogue duplex modes: `barge_in` and `full_duplex` config toggles — ✅ FIXED

**Resolution:** Added `barge_in: bool = True` and `full_duplex: bool = True`
to `DialogueSettings` (`src/config.py`) and threaded both through
`src/cli/dialogue.py`. The speak callback conditionally constructs the
`barge_event` (None = no barge-in → `cancel=None` to `play_tts_streaming`)
and the `tts_active` gate (None = full-duplex → listener skips the gate
entirely). Half-duplex is enforced by having `_make_speak_callback`
set/clear `tts_active` around the TTS call in a `try/finally` so
exceptions still release the gate, and `_listener_loop` poll-waits on
`tts_active.is_set()` via `shutdown.wait(0.05)` before each `record()`.
Six new tests in `TestDuplexModes` cover the matrix: barge_in on/off
(cancel= forwarding), half-duplex (tts_active set during play + cleared
on exception + listener waits for the gate), and full-duplex regression
(listener records immediately with `tts_active=None`). 162/162 tests
pass. Matrix documented in `speak-dialogue.md`.

**Severity:** Feature (unblocks loopback/self-dialogue workflows and open-speaker
setups without headphones).

**Context:** P7 landed barge-in via a shared `barge_event` that cancels in-flight
TTS the moment VAD fires. That's correct for headphones but pathological on open
speakers — the mic hears the speakers, VAD trips, TTS cancels itself. Separately,
a valid use case exists for *intentionally* routing TTS output back through the
mic: agent-talks-to-itself / self-dialogue / loopback validation.

Both scenarios (and the middle-ground "speaker-safe half-duplex") fall out of
two orthogonal config flags:

| `barge_in` | `full_duplex` | Behavior                                                |
|------------|---------------|---------------------------------------------------------|
| true       | true          | **Current (P7) behavior** — full-duplex with interrupt. Good on headphones. |
| true       | false         | **Speaker-safe** — mic paused while TTS plays; interrupt applies to the *next* utterance only. No echo feedback possible. |
| false      | true          | **Loopback / self-dialogue** — mic always hot, TTS always finishes, feedback loop is the *intended path*. |
| false      | false         | **Walkie-talkie** — strict half-duplex, no interrupt. Simplest and most predictable. |

**Scope:**

1. Add to `src/config.py` `DialogueSettings`:
   - `barge_in: bool = True`
   - `full_duplex: bool = True`
2. Thread both into `src/cli/dialogue.py`:
   - `barge_in=False` → pass `cancel=None` to `play_tts_streaming` (or don't
     construct the `barge_event` at all).
   - `full_duplex=False` → the listener loop skips `recorder.record()` while
     `inference_lock` is held by TTS. Either poll on a `tts_active: Event`
     (set by the speak callback around the `play_tts_streaming` call) or
     have the listener acquire-then-release the lock as a gate before each
     record. The `Event` approach is cleaner — no lock churn in the hot path.
3. Tests in `tests/test_cli_subcommands.py`:
   - Parametrized matrix over the four combinations, asserting:
     - `barge_in=False` → `play_tts_streaming` is called with `cancel=None`.
     - `full_duplex=False` → listener blocks on `tts_active` before each record.
     - Both true = P7 behavior unchanged (regression guard).
     - Both false = mic never records while TTS plays AND TTS never cancels.
4. Document the matrix in `speak-dialogue.md` alongside the existing threading
   model section.

**Why it matters:**

- Without this, open-speaker setups are effectively unusable (P7's own
  follow-up note). The `full_duplex=false` toggle is the right answer to
  the echo problem — a gate, not a ducking mixer.
- The `barge_in=false` toggle is the *motivating* request: it makes the
  feedback loop a feature instead of a bug, and opens a real workflow
  (agent speaks, transcribes its own output, chains). Removing that path
  under the "fix echo" banner would have been a loss.
- The two flags are orthogonal and simple — no mode enum, no state machine.

**Out of scope:** audio ducking / mixer gain control, device-level echo
cancellation, any form of automatic headphone-vs-speaker detection. Those
are real problems but they're all harder than a one-line config flag and
belong to a separate design pass.

**Files:** `src/config.py`, `src/cli/dialogue.py`, `tests/test_cli_subcommands.py`,
`speak-dialogue.md`
